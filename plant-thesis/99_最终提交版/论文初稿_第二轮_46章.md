# 第四章　系统总体设计

## 4.1　系统设计目标

本系统的总体设计围绕"植物科普可视化交互"这一核心目标展开，旨在将多源植物数据转化为面向普通用户可浏览、可分析、可交互的 Web 系统。系统设计目标具体包括以下四个方面：

第一，构建统一的多源植物数据组织体系。系统需整合 WCVP、IUCN、WDPA、WGSRPD 等来源的权威植物数据，对植物基础信息、分类结构、地理分布、濒危状态与图像资源进行统一建模，为前台展示与分析提供数据基础。

第二，设计清晰的前后端分离架构。前端专注于页面渲染与用户交互，后端专注于业务逻辑与数据查询，通过标准 RESTful API 完成数据通信，提高系统的可维护性与可扩展性。

第三，围绕植物科普场景设计核心功能模块，使各页面模块之间具有良好的业务衔接关系，体现"植物浏览→分类认知→分布分析→学习测验"的科普引导路径。

第四，以可视化表达为重点设计方向，将植物分类结构与地理分布信息以图形化方式呈现，使用户能够通过交互探索理解抽象的植物学知识。

## 4.2　系统总体架构设计

### 4.2.1　整体分层架构

本系统采用前后端分离的分层架构，整体请求链路为：

**浏览器（React SPA）→ API 接入层（Express Router + 中间件）→ 业务逻辑层（Service）→ 数据访问层（Sequelize / Raw SQL）→ 数据层（MySQL 8.0）**

【图4.1 系统总体架构图，待插入】

各层职责划分如下：

**前端展示层**：采用 React 19 + TypeScript + Vite 构建单页应用（SPA），共实现 9 个核心页面组件。前端通过 `api.ts` 统一封装所有 HTTP 请求，内置 GET 请求本地缓存、并发请求去重（inflight 去重）和缓存局部失效机制，避免相同请求在短时间内重复发出。页面跳转通过 URL 参数（page、plantId、search、sort 等）与状态同步实现，无需引入专用路由库。

**API 接入层**：由 Express Router 负责 URL 路由分发。请求在到达 Controller 前，依次经过 CORS 中间件（跨域控制）、JWT 身份验证中间件（仅针对需要鉴权的接口）和请求参数校验中间件（格式验证），确保请求的合法性。

**业务逻辑层**：共 13 个 Service 文件，承载各业务域的核心逻辑，包括植物数据组装、分类树递归构建、搜索排序策略、预警等级计算、测验评分等。Controller 层仅负责 HTTP 请求/响应的格式处理，业务逻辑均下沉至 Service 层，保证职责清晰。

**数据访问层**：以 Sequelize ORM 模型查询为主，对复杂聚合统计场景辅以 `sequelize.query()` 原生 SQL 查询，在开发效率与查询灵活性之间取得平衡。

**数据层**：MySQL 8.0，共 29 张数据表，覆盖植物主数据、分类学、保护濒危、地理分布、用户、学习与媒体七大业务域。

### 4.2.2　数据工程管道架构

系统在主业务架构之外，构建了一套独立运行的数据工程管道，负责从外部权威数据源采集、清洗、转换和导入植物数据。管道以脚本形式实现，与主服务解耦，可独立运行而不影响在线服务。

【图4.2 数据工程管道流程图，待插入】

管道主要步骤如下：

**步骤一：WCVP 分类骨架导入。** 通过 `import-wcvp.js` 脚本读取 WCVP CSV 数据文件，按照分类等级（family → genus → species）分批导入 `taxa` 表，仅保留 `taxon_status = 'Accepted'` 的接受名记录，以 WCVP 接受名作为系统学名的权威基准[@govaerthew2021]。

**步骤二：分类链修复与质量清洗。** 通过 `repair-taxonomy-chain.js` 自动修复父节点引用缺失问题，通过 `cleanup-duplicate-taxa.js` 合并重复的分类节点，保证 `taxa` 表中每个科、属、种仅有唯一记录。

**步骤三：中文名四级优先级映射。** 针对植物学名与中文名对应关系缺失的问题，系统设计了一套基于优先级队列的自动映射策略（详见第五章 5.1.2 节）。

**步骤四：IUCN 红色名录导入与预警生成。** 通过 `import-iucn-redlist.js` 导入 IUCN 数据，并在导入时自动根据物种等级变化生成濒危预警记录[@iucnredthr2023]。

**步骤五：WDPA 保护地与 WGSRPD 分布数据接入。** 分别导入全球保护地边界数据与 WGSRPD 三级地理分区分布数据，构建地理可视化的数据基础。

**步骤六：iPlant 详情采集。** 通过接口逆向分析方法，批量获取植物形态特征、生态习性和功用价值内容，写入 `plant_detail` 表（详见第五章 5.1.3 节）。

**步骤七：就绪度校验。** 通过 `check-data-readiness.js` 验证中文名覆盖率、封面图覆盖率和分类链完整性等关键指标，确认达标后方对外提供服务。

## 4.5　数据库设计

### 4.5.1　概念模型设计

系统数据库共包含 29 张数据表，按业务域可分为七大领域：植物主数据域、分类学域、保护与濒危域、地理分布域、用户域、学习域和媒体域。

【图4.8 核心数据库 ER 图，待插入（建议通过 MySQL Workbench Database→Reverse Engineer 逆向生成）】

核心实体与关联关系如下：
- `taxa` 表通过 `parent_id` 字段自关联，构成有根树结构，支持递归遍历科—属—种三级层次；
- `plants` 表通过 `taxon_id` 与 `taxa` 关联（N:1），每个物种对应一个分类节点；
- `plant_detail` 表与 `plants` 表为一对一关联，存储植物的详细描述字段；
- `plant_media` 表作为连接表，实现 `plants` 与 `media_assets` 之间的多对多关联（同一图片资源可被多个植物复用）；
- `plant_distributions` 表记录植物与 WGSRPD 三级地理分区的对应关系（N:1 → plants，N:1 → wgsrpd_regions）；
- `threatened_species` 和 `redlist_alerts` 共同构成濒危预警数据链；
- `favorites` 和 `browse_events` 以 `user_id` 和 `plant_id` 为核心关联字段，分别记录收藏行为与浏览行为；
- 测验系统形成三层链式结构：`quizzes → questions → question_options`（1:N 链）；
- 答题记录形成两层结构：`quiz_attempts → attempt_answers`，关联用户与测验。

# 第五章　系统功能实现

## 5.1　数据层实现

### 5.1.1　多源数据融合导入

**WCVP 分类数据导入**

`import-wcvp.js` 脚本读取 WCVP CSV 文件，按以下逻辑完成分类节点导入：首先依据 `wcvp_plant_name_id` 字段判断记录是否已存在，避免重复导入；其次解析 `family` 和 `genus` 字段，在导入种（species）时自动查找并关联已存在的属（genus）父节点；最后仅保留 `taxon_status = 'Accepted'` 的接受名，过滤掉同物异名（synonym）记录，以保证分类数据的规范性[@govaerthew2021]。

**IUCN 红色名录导入与预警生成**

`import-iucn-redlist.js` 在导入濒危物种数据的同时，自动执行预警生成逻辑：将当前 IUCN 等级与历史等级进行比对，根据变化方向判断 `changeType`（`upgraded` / `downgraded` / `new_addition` / `new_assessment`），并据此计算 `alertLevel`（`high` / `medium` / `low`）。关键判断逻辑如下：

```javascript
// 节选自 import-iucn-redlist.js
function determineAlertLevel(prevCategory, currCategory, changeType) {
  if (currCategory === 'CR' && changeType === 'upgraded') return 'high';
  if (['EN', 'CR'].includes(currCategory)) return 'high';
  if (currCategory === 'VU') return 'medium';
  return 'low';
}
```

**WDPA 与 WGSRPD 数据接入**

WDPA 保护地数据通过 `import-wdpa.js` 脚本批量导入 `protected_areas` 表，记录保护地名称、国家代码（ISO3）、IUCN 保护类别和地理边界等信息。WGSRPD 三级地理分区分布数据通过 `load-distribution-and-build.js` 写入 `plant_distributions` 表，建立植物与地理分区之间的对应关系，为前端热力图渲染提供数据基础。

### 5.1.2　中文名智能映射策略

针对植物学名与中文名对应关系缺失的问题，系统设计了一套基于优先级队列的自动映射策略，依次尝试以下四个数据来源：

【图5.1 中文名四级优先级映射流程图，待插入】

**第一优先级：iPlant 植物智数据库。** 通过 `fetch-chinese-names.js` 脚本批量查询 iPlant API，命中结果直接写入 `taxa` 表，标记 `translation_source = 'iplant'`，准确率最高。

**第二优先级：GBIF/Wikidata API 验证。** 对 iPlant 未覆盖的物种，通过 `fill-cn-template-gbif-strict.js` 和 `fill-cn-template-wikidata-strict.js` 分别查询 GBIF 和 Wikidata，仅采用置信度达标（GBIF match confidence > 0.9）或有明确中文标签的结果[@gbifsecbac2022]。

**第三优先级：学名模板推断。** 通过 `fill-cn-template-manual.js` 脚本，根据上级分类节点的已有中文名，按照"属名＋种加词语义"规则构造候选中文名，适用于属级以上分类节点。

**第四优先级：人工审核队列。** 经前三级策略仍无法获得中文名的物种，通过 `export-human-review-queue.js` 导出为待审核列表，由人工完成最终确认。

通过上述四级策略，系统在批量处理阶段对有中文名的植物保持了较高覆盖率，显著提升了系统对中文用户的友好度。

### 5.1.3　植物详情数据自动采集

系统通过对 iPlant 植物智平台内部接口的逆向分析，实现了植物形态特征、生态习性和功用价值三类详情内容的批量自动采集。采集流程分为以下步骤：

**步骤一：获取物种编号（spno）。** 对每个植物学名构造 URL 访问 `https://www.iplant.cn/info/{学名}`，通过正则表达式从返回 HTML 的 JavaScript 变量中提取物种编号：`var spno = "18429"`。

**步骤二：并行请求三类描述数据。** 以 spno 为参数，通过 POST 请求调用 `/ashx/plantinfo.ashx` 接口，分别传入 `typeid=11`（形态特征）、`typeid=12`（生态习性）、`typeid=20`（功用价值），三次请求并行发出。

**步骤三：Base64 解码与格式化。** 接口返回的 JSON 数组中，各子项的 `desc` 字段为 Base64 编码的中文文本，解码后按"【子项名称】描述文本"格式拼接：

```javascript
// 节选自 fill-plant-detail-from-iplant.js
function decodeItems(items) {
  return items
    .filter(item => item.hasdesc && item.desc)
    .map(item => {
      const text = Buffer.from(item.desc, 'base64').toString('utf8').trim();
      return `【${item.t}】${text}`;
    })
    .join('\n');
}
```

**步骤四：写入数据库。** 采用 `INSERT ... ON DUPLICATE KEY UPDATE` 语句写入 `plant_detail` 表，并通过条件判断确保已有手工录入的内容不被覆盖。采集过程设置 1.5～2 秒的请求间隔以避免触发平台频率限制，并通过进度文件记录已处理的 plant_id，支持中断后断点续传。

## 5.2　后端核心模块实现

### 5.2.1　分类树递归查询实现

分类树采用按需加载（懒加载）策略，避免一次性加载全量分类节点（约 39 万条）带来的性能问题。`taxonomyService.js` 的实现逻辑如下：

前端首次请求顶层节点时，不传 `parentId` 参数，服务端查询 `parent_id IS NULL` 的节点列表并返回，同时对每个节点计算 `has_children` 标志（判断是否存在子节点）。当用户点击展开某一节点时，前端携带该节点 ID 请求其子节点列表，服务端执行 `WHERE parent_id = ?` 查询并返回。这种方式使每次加载的数据量仅为一级子节点，响应速度可控。

前端 `Classification.tsx` 通过 `expandedIds`（`Set<string>`）状态管理展开节点集合，通过 `updateNodeChildren()` 递归函数将新加载的子节点插入树状数据结构的对应位置，实现状态的局部更新而非整体重建。

### 5.2.2　植物检索服务实现

`searchService.js` 实现了多字段联合搜索与多维度排序功能。关键词搜索对 `chinese_name` 和 `scientific_name` 字段分别进行 `LIKE 'keyword%'` 前缀匹配（利用已建前缀索引），并通过 `CASE WHEN` 表达式对前缀命中赋予更高权重。排序策略支持"最新收录"（按 `plants.id DESC`）、"热度"（关联 `plant_popularity_daily` 表计算近期热度分）和"名称首字母"（按 `scientific_name ASC`）三种模式。分页通过 `LIMIT` 和 `OFFSET` 实现，响应结果包含 `total` 总数供前端计算总页数。

### 5.2.3　濒危预警服务实现

`redlistService.js` 负责预警记录的查询与用户状态管理。预警的已读/忽略状态存储于独立的 `redlist_alert_user_state` 表（以 `user_id` 和 `alert_id` 为联合主键），而非写回预警主表 `redlist_alerts`。这一设计的优点在于：同一条预警对不同用户可以维护各自独立的状态，预警主表不随用户操作产生写放大，查询效率更高。

### 5.2.4　学习测验服务实现

测验系统采用三层数据结构：`quizzes`（测验）→ `questions`（题目）→ `question_options`（选项），通过 Sequelize 的嵌套 `include` 查询一次性加载完整测验数据。用户提交答题时，服务端对每道题逐一对比用户选择与正确选项，计算总分后生成 `QuizResult` 响应。答题过程的原始数据写入 `attempt_answers` 表（每题一条记录），`quiz_attempts` 表记录本次答题的汇总信息（总分、完成时间），供学习历史查询使用。

## 5.3　前端核心模块实现

### 5.3.1　全球分布热力图实现

全球分布热力图是本系统可视化实现的重点，基于浏览器原生 SVG 技术实现，不依赖第三方图表库。

**数据获取：** 前端调用 `getHeatmapMap(180, presenceFilter)` 接口，获取 WGSRPD 三级地理分区的 GeoJSON 边界数据（含各分区物种数量统计）。

**地理投影：** `coordinatesToPath()` 函数将 GeoJSON 多边形坐标数组（经度、纬度）通过线性映射转换为 SVG 画布像素坐标，公式为：

```
x = (lng - minLng) / (maxLng - minLng) × MAP_WIDTH
y = (maxLat - lat) / (maxLat - minLat) × MAP_HEIGHT
```

其中画布尺寸 MAP_WIDTH = 1200，MAP_HEIGHT = 620，地理范围为经度 [-180, 180]，纬度 [-60, 85]。

**SVG 渲染与颜色映射：** 将各分区的 GeoJSON 多边形转换为 SVG `<path>` 元素，按各分区物种数量在预定义绿色渐变色板（`['#dcfce7', '#86efac', '#4ade80', '#16a34a', '#166534']`）中映射填充颜色，物种数量越多，颜色越深。

**交互实现：** 通过 `onMouseEnter` / `onMouseLeave` 实现区域悬停高亮；通过 `onClick` 实现区域选中，选中后根据该分区的 `bbox`（边界框）计算 SVG `viewBox` 属性，实现视口自动缩放至选中区域；侧边栏同步请求该区域植物列表并展示。此外，通过 `presenceFilter`（全部/原生/引种）状态变量触发数据重新请求，实现分布类型筛选。

**第一，完成了多源植物数据的融合接入与组织建模。** 系统以 WCVP 为分类学骨架，整合 IUCN 红色名录、WDPA 保护地数据库和 WGSRPD 地理分区标准，并通过对 iPlant 植物智内部接口的逆向分析实现了详情内容的批量自动采集，构建了包含 29 张数据表、覆盖七大业务域的植物知识数据库[@govaerthew2021][@iucnredthr2023]。

**第二，设计并实现了中文名四级优先级映射策略。** 针对植物学名与中文名对应关系缺失的问题，系统依次通过 iPlant 数据库、GBIF/Wikidata API、学名模板推断和人工审核队列四个层次完成中文名的自动补全，提升了系统对中文用户的知识呈现能力[@gbifsecbac2022]。

**第三，基于 SVG 技术实现了全球植物分布热力图。** 系统以 WGSRPD 三级地理分区为空间单元，采用浏览器原生 SVG 技术实现了全球分布热力图，支持区域点击缩放、地图与表格视图切换及原生/引种分布筛选，在不依赖第三方图表库的前提下实现了可交互的地理可视化功能[@munznervis2014]。

**第四，构建了完整的学习交互闭环。** 系统在植物浏览基础上引入浏览行为记录、植物收藏、推荐测验和答题反馈机制，使用户能够在科普浏览之余进行知识巩固，形成"浏览→收藏→测验→反馈→复习"的完整学习路径，提升了系统的参与感与连续性。

**第五，通过系统测试验证了各功能的可用性。** 功能测试表明各核心模块运行稳定，工程验证通过了后端单元测试、前端类型检查和生产构建验证。系统整体满足植物科普可视化展示与交互学习的设计目标，可用于毕业设计答辩演示与实际使用。

本文存在以下局限性：一是系统当前植物详情内容覆盖率尚不完整，部分物种的形态、生境与用途字段仍为空；二是现有测验题库数量有限，学习测验功能的实用性依赖后续数据补充；三是系统未进行高并发与大流量场景下的深度性能测试，在工程化部署层面还有进一步完善的空间。

## 展望

基于当前系统的实现边界与已识别的局限性，未来可从以下几个方向继续优化。

**第一，持续扩展植物详情内容覆盖。** 当前 iPlant 采集管道在中国植物范围内命中率约 30%～44%，后续可继续运行采集脚本扩大覆盖，同时可尝试接入中国植物志全文（Flora of China）数字化数据，提升植物形态与生态信息的丰富度。

**第二，增加公民科学观测数据模块。** 参考 iNaturalist 的社区观测模式，可在系统中增加用户实地观测记录上传功能，允许用户提交拍摄图片和地理位置信息，形成本地化的公民科学植物观测数据集，增强系统的社区互动属性[@heeratotou2010]。

**第三，引入植物图像识别辅助功能。** 可结合深度学习图像分类模型（如基于卷积神经网络的植物识别模型），为用户提供上传照片辅助鉴定植物种类的功能，进一步降低植物知识探索的门槛。

**第四，优化移动端交互体验。** 当前系统以桌面端为主要目标场景，热力图缩放与分类树展开在移动端触控操作上存在改进空间。后续可基于 PWA（渐进式 Web 应用）技术优化移动端体验，并针对触控操作习惯调整热力图与分类树的交互逻辑。

**第五，提升搜索引擎能力。** 当前检索基于 MySQL LIKE 前缀查询，在数据规模持续增长的情况下存在性能瓶颈。后续可引入 Elasticsearch 等全文搜索引擎，支持拼音输入搜索、同义词扩展和按相关度排序，进一步提升植物检索的精准性与响应速度。
