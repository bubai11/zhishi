# 第一章　绪论

## 1.1　研究背景与意义

植物是生态系统的重要组成部分，也是生物多样性保护与自然科普教育中的关键对象。据世界栽培植物词汇表（World Checklist of Vascular Plants，WCVP）统计，全球已记录维管植物超过 35 万个接受种，涉及名称条目逾 1100 万条[@govaerthew2021]。与此同时，国际自然保护联盟（IUCN）红色名录评估结果显示，约 21% 的已评估植物物种面临不同程度的灭绝威胁[@iucnredthr2023]。在此背景下，面向公众开展植物多样性科普教育，具有重要的生态保护与社会意义。

随着互联网技术和数据可视化技术的不断发展，构建面向普通用户的植物科普 Web 系统已成为可行路径。通过整合权威开放数据，并以图形化、结构化、交互化的方式加以呈现，用户不仅能够检索到植物基本信息，还能够理解植物的分类关系、地理分布规律和保护状态，从而形成更完整的植物知识认知[@munznervis2014]。

然而，现有植物信息系统仍存在以下几方面不足。第一，数据来源分散，WCVP、IUCN 红色名录、全球保护地数据库（WDPA）等权威数据集各自独立，缺乏统一的接入与组织。第二，现有主流平台（如 GBIF、Plants of the World Online）以英文界面和专业使用场景为主，中文科普表达和本土化知识组织能力不足，对普通中文用户存在较高使用门槛。第三，许多系统停留在静态信息展示层面，缺乏对分类结构、空间分布和统计结果的可视化呈现，以及与用户学习行为相结合的交互反馈机制。

基于上述问题，本文设计并实现了一个面向植物科普学习的知识可视化与交互系统。系统以多源数据融合作为基础，以植物知识可视化作为主要表达方式，以学习交互闭环作为补充设计，在知识准确性、展示直观性和用户参与性之间取得平衡。

本系统的研究意义主要体现在以下两个层面：在技术层面，探索多源异构植物数据（WCVP、IUCN、WDPA、WGSRPD、iPlant）在 Web 系统中的整合建模与可视化表达方法；在应用层面，为中文植物科普传播、植物知识学习和生态保护宣传提供一个可演示、可交互、可扩展的数字化平台。

## 1.2　国内外研究现状

### 1.2.1　国内研究现状

国内在植物信息数字化领域已积累了一定的基础。中国植物志数字化项目（Flora of China Online）是目前国内最权威的植物志数字化成果，收录了中国维管植物约 31000 种[@reftodo001]。iPlant 植物智平台由中国科学院植物研究所主导，提供了较为丰富的中文植物信息，涵盖物种描述、形态图片和分布信息，是国内较具代表性的植物信息平台。上述平台在数据积累和专业性方面较强，但大多偏向资料检索与专业参考，面向普通用户的可视化交互能力相对有限。

在学术研究方面，国内学者围绕植物知识图谱构建、植物图像识别、生态数据管理和科普平台开发等方向开展了较多探索[@reftodo002]。部分研究重视数据建模与知识抽取，部分研究聚焦深度学习图像分类与智能推荐，还有部分研究针对教育场景构建了学习平台。然而，专门围绕"植物科普可视化交互"进行系统设计的研究相对较少，尤其是将多源数据融合、分类树交互、地理分布可视化和用户学习行为统一集成到同一系统的工作仍有较大提升空间。

### 1.2.2　国外研究现状

国外在植物与生物多样性信息平台方面发展较早。GBIF（全球生物多样性信息机构）是目前全球规模最大的生物多样性开放数据平台，聚合了来自全球数千个机构的超过 20 亿条物种观测记录，并提供开放 API 供二次开发[@gbifsecbac2022]。Plants of the World Online（POWO）由英国皇家植物园邱园（Kew Gardens）维护，以 WCVP 为数据基础，提供了较为完整的植物分类学信息。iNaturalist 是面向公众的公民科学平台，通过用户上传照片与社区鉴定相结合的方式，已成为全球活跃度较高的自然观察社区之一[@heeratotou2010]。

然而，上述国外平台大多以英文界面和专业场景为主，中文支持有限，且交互设计偏向数据检索与专业引用，面向科普传播的可视化叙事与学习交互功能相对不足，难以完全满足国内普通用户的科普需求。

### 1.2.3　现有系统对比与本文定位

综合国内外研究现状，现有系统在数据积累与专业查询方面已取得较多成果，但在以下方面仍存在不足：其一，多源权威数据的系统性整合程度不高，植物分类、分布、濒危状态与图像数据往往分散于不同平台；其二，面向普通用户的中文科普表达不足，主流系统偏向专业工具化；其三，植物知识的可视化深度不够，缺少对分类结构与空间分布的整体化展示；其四，知识浏览与用户交互之间的联系偏弱，系统多以静态信息查看为主。

基于上述分析，本文的研究定位为：以植物科普传播为目标，以多源数据融合为基础支撑，以植物知识可视化和交互式浏览为重点设计方向，构建一个兼顾知识展示、分类认知、区域分析和轻量学习交互的综合性 Web 系统，在数据权威性与用户友好性之间寻求平衡。

**表 1.1　本系统与现有平台对比**

| 对比维度 | 本系统 | GBIF | iPlant 植物智 | POWO |
|---|---|---|---|---|
| 数据来源 | WCVP/IUCN/WDPA/WGSRPD/iPlant 多源融合 | 观测记录聚合 | 国内采集与整理 | WCVP 单一来源 |

| 开放程度 | 本地部署 | 完全开放 API | 部分开放 | 有限开放 |

## 1.3　主要研究内容

本文围绕系统设计与实现，主要开展以下五个方面的工作。

**第一，多源植物数据的接入、清洗与融合。** 系统以 WCVP 数据为分类学骨架，批量导入科、属、种三级分类节点，并通过专用脚本完成分类链修复与重复数据清理。在此基础上，设计了一套基于四级优先级的中文名映射策略：依次尝试 iPlant 植物智数据库、GBIF/Wikidata API 验证、学名模板推断，失败后进入人工审核队列。此外，系统通过对 iPlant 内部接口的逆向分析，实现了植物形态特征、生态习性和功用价值三类内容的批量自动采集。

**第二，面向植物科普场景的数据建模。** 根据植物知识展示与分析需求，对植物、分类、图片、分布、濒危、题库、用户行为等核心数据进行结构化设计，构建了包含 29 张数据表、覆盖植物主数据、分类学、保护濒危、地理分布、用户、学习与媒体七大业务域的关系型数据库。

**第三，植物科普可视化的设计与实现。** 围绕植物分类结构，设计并实现了支持懒加载递归展开的科—属—种三级分类树；围绕植物地理分布，基于 WGSRPD 三级地理分区标准，以浏览器原生 SVG 技术实现了全球植物分布热力图，支持地图与表格视图切换及原生/引种分布筛选。

**第四，濒危预警与保护地信息整合。** 系统整合 IUCN 红色名录与 WDPA 保护地数据，实现了面向注册用户的濒危预警推送功能，包括预警等级分类、预警状态管理，以及保护地的多维度检索与统计。

**第五，学习交互闭环的设计与实现。** 系统设计了"浏览记录→植物收藏→知识测验→答题反馈→用户成就"的完整学习支持体系，以数据驱动的方式激励用户在科普浏览基础上持续深化知识理解。

## 1.4　论文组织结构

本文共分为六章，各章内容安排如下：

第一章为绪论，介绍研究背景与意义、国内外研究现状、主要研究内容及论文组织结构。

第二章为相关技术分析，介绍系统前端框架（React 19、TypeScript、Vite、Tailwind CSS、SVG 可视化）、后端框架（Node.js、Express、Sequelize、JWT）、数据库（MySQL 8.0）及多源开放数据标准（WCVP、IUCN、WDPA、WGSRPD）。

第三章为系统需求分析，从功能需求与非功能需求两方面分析系统建设目标，并通过用例分析说明核心业务流程。

第四章为系统总体设计，介绍系统分层架构、功能模块划分、数据工程管道设计及数据库概念设计。

最后为结论与展望，总结全文工作并对系统未来的优化方向进行展望。

# 第二章　相关技术分析

## 2.1　前端相关技术

### 2.1.1　React 19 与组件化开发

React 是由 Meta（原 Facebook）开源维护的前端 JavaScript 框架，采用组件化架构与声明式 UI 编程范式。本系统采用 React 19 版本，其引入的并发渲染（Concurrent Rendering）机制允许根据用户交互动态调整组件渲染优先级，对于数据量较大的植物列表和分类树场景，能够有效提升页面响应流畅度。

React 以组件（Component）为基本单元，每个组件通过 Hooks 机制管理自身状态（State）与副作用（Effect）。本系统前端共实现了 9 个核心页面组件：`Home`（首页）、`Library`（植物库）、`PlantDetail`（植物详情）、`Classification`（分类浏览）、`Analysis`（数据分析）、`LearningCenter`（学习中心）、`UserProfile`（用户中心）、`Navbar`（导航栏）和 `Footer`（底栏）。各组件职责清晰、相互独立，便于功能迭代与维护。

选用 React 的主要理由在于其成熟的生态体系、完善的 TypeScript 支持，以及 Hooks 状态管理机制对复杂植物数据交互场景（如分类树懒加载、热力图区域联动）的良好适配[@reftodo003]。

### 2.1.2　TypeScript 类型系统

TypeScript 是微软开发的 JavaScript 超集，在 JavaScript 基础上引入了静态类型系统、接口定义和枚举等特性。在前后端分离的系统架构中，TypeScript 能够在编译阶段发现接口数据结构不一致等问题，显著降低运行时错误风险。

本系统前端在 `types.ts` 中统一定义了全部核心数据类型，包括 `PlantCard`（植物列表卡片）、`PlantDetailResponse`（植物详情响应）、`TaxonomyNode`（分类节点）、`Quiz`（测验）、`Alert`（预警）等。类型定义确保前端组件与后端 API 之间的数据结构契约在开发阶段得到静态校验，减少了接口联调阶段的排查成本。

### 2.1.3　Vite 6 构建工具

Vite 是由 Vue.js 作者尤雨溪主导开发的新一代前端构建工具。在开发模式下，Vite 基于浏览器原生 ES Module（ESM）按需编译，仅在浏览器请求时转换对应模块，启动速度相比传统 Webpack 的全量打包方式有显著提升。在生产构建阶段，Vite 使用 Rollup 进行代码打包与 Tree-Shaking 优化，减小产物体积。

本系统使用 Vite 6 版本，开发阶段支持热模块替换（HMR），修改代码后页面局部更新无需整体刷新，有效缩短了前端调试的反馈周期。

### 2.1.4　Tailwind CSS 4 原子化样式框架

Tailwind CSS 是一个以"原子化 CSS"（Utility-First CSS）为核心理念的样式框架，提供大量细粒度的功能类，允许开发者直接在 JSX 中通过类名组合实现复杂样式，无需维护大量自定义 CSS 文件[@reftodo004]。

本系统使用 Tailwind CSS 4 版本，结合其即时编译（JIT）引擎，仅将实际使用的样式类编译进最终产物，有效控制了 CSS 体积。系统界面采用统一的绿色系色彩体系，通过 Tailwind 的响应式前缀实现桌面端的自适应布局。

### 2.1.5　SVG 自定义地图可视化

系统的全球植物分布热力图基于浏览器原生 SVG 技术实现，未引入 ECharts、Leaflet 等第三方图表库。选择自行实现的主要原因是：专用可视化库通常体积较大，且对于基于地理分区着色这类需求往往存在定制困难；相比之下，直接操作 SVG 元素能够以较小的代码量实现精确的交互控制。

实现原理如下：后端通过 `/api/wcvp-analytics/heatmap-map` 接口返回 WGSRPD 三级地理分区的 GeoJSON 数据，前端 `coordinatesToPath()` 函数将 GeoJSON 多边形坐标通过线性地理投影（经纬度→像素坐标映射）转换为 SVG `<path>` 元素的 `d` 属性字符串。各分区依据物种数量在预定义的绿色渐变色板（`['#dcfce7', '#86efac', '#4ade80', '#16a34a', '#166534']`）中映射填充颜色。交互方面，通过 `onMouseEnter` / `onClick` 事件分别实现区域悬停高亮与选中后视口缩放至对应边界框（bbox）的效果。

## 2.2　后端相关技术

### 2.2.1　Node.js 与 Express 框架

Node.js 是基于 Chrome V8 引擎构建的 JavaScript 服务端运行时，采用单线程事件循环与异步非阻塞 I/O 模型。该模型在处理大量并发的 API 请求时，不会因 I/O 等待而阻塞线程，适合本系统植物数据查询、图片资源访问等数据密集型场景[@reftodo005]。

Express 是基于 Node.js 的轻量级 Web 框架，提供路由管理、中间件机制与请求/响应封装。本系统后端采用 Express 4.x，按照 MVC 分层组织代码：`routes/` 负责 URL 路由分发，`controllers/` 负责 HTTP 请求与响应处理，`services/` 承载核心业务逻辑，`models/` 定义 Sequelize 数据模型。系统共实现了 11 个路由模块（plants、taxa、search、redlist、protected-areas、quizzes、browse-events、favorites、user、media-assets、wcvp-analytics），覆盖全部业务域。

### 2.2.2　Sequelize ORM

Sequelize 是 Node.js 生态中功能完整的对象关系映射（ORM）框架，支持 MySQL、PostgreSQL、SQLite 等主流关系型数据库。ORM 框架将数据库表映射为 JavaScript 对象，使开发者可以用面向对象方式操作数据库，同时提供关联查询（Associations）等高级特性。

本系统使用 Sequelize 6，在 `models/` 目录下为 29 张数据表各定义对应模型，并在 `associations.js` 中集中管理模型间的关联关系（`belongsTo`、`hasOne`、`hasMany`、`belongsToMany`）。对于复杂的聚合统计和性能敏感场景，系统在 Sequelize 基础上辅以 `sequelize.query()` 原生 SQL 查询，在开发效率与查询灵活性之间取得平衡。

### 2.2.3　JWT 无状态身份认证

JSON Web Token（JWT）是一种开放标准（RFC 7519），用于在网络应用间安全传递声明信息[@reftodo006]。JWT 由 Header、Payload 和 Signature 三部分组成，服务端签发 Token 后下发客户端，客户端在后续请求中将 Token 置于 `Authorization` 请求头，服务端通过验证签名完成身份鉴权，无需在服务端维护 Session 状态，适合前后端分离的无状态架构。

本系统采用 JWT 实现用户身份认证，配合 bcryptjs 对用户密码进行哈希存储。在需要鉴权的接口（收藏、浏览记录、测验提交、预警管理等）前置 JWT 中间件，对未携带有效 Token 的请求返回 401 状态码拦截。

## 2.3　数据库相关技术

### 2.3.1　MySQL 8.0

本系统使用 MySQL 8.0 作为关系型数据库。选择 MySQL 的主要理由包括：成熟的事务支持（InnoDB 引擎）、完善的外键约束机制、对 JSON 字段类型的原生支持，以及与 Sequelize ORM 的良好集成。

系统利用 MySQL 8.0 的 JSON 字段类型在 `plant_detail.extra` 列中存储植物的扩展属性，避免了为少量扩展字段修改表结构的代价，同时通过 InnoDB 的外键约束保障核心数据（如植物与分类节点、植物与媒体资产）之间的参照完整性。

### 2.3.2　数据库索引优化

系统针对高频访问的查询路径进行了专项索引设计：在 `plants` 表的 `chinese_name`、`scientific_name` 字段建立前缀索引，支持关键词检索场景下 `LIKE 'keyword%'` 形式的前缀匹配；在 `taxa` 表的 `parent_id` 字段建立索引，优化分类树递归查询的子节点加载性能；在 `plant_distributions` 表的 `wgsrpd_level3_code` 字段建立索引，支持按地理区域高效筛选植物分布数据。上述索引通过专用脚本 `ensure-performance-indexes.js` 统一创建与维护，确保在数据迁移后自动重建。

## 2.4　多源开放数据标准

### 2.4.1　WCVP（世界栽培植物词汇表）

WCVP（World Checklist of Vascular Plants）由英国皇家植物园邱园（Kew Gardens）维护，是目前全球最权威的维管植物名称数据库，收录超过 35 万个接受种的分类信息[@govaerthew2021]。系统以 WCVP 发布的 CSV 数据文件为输入，通过 `import-wcvp.js` 脚本批量导入科、属、种三级分类节点，仅保留 `taxon_status = 'Accepted'`（接受名）的记录，以 WCVP 接受名作为系统学名的权威基准。

### 2.4.2　IUCN 红色名录

IUCN 红色名录是全球最具权威性的物种濒危状况评估体系，将物种按灭绝风险分为灭绝（EX）、野外灭绝（EW）、极危（CR）、濒危（EN）、易危（VU）、近危（NT）、无危（LC）七个等级[@iucnredthr2023]。系统通过 `import-iucn-redlist.js` 脚本接入 IUCN 数据，在导入时根据物种等级变化（前次评估等级与当前等级对比）自动判断变化类型（升级/降级/新增），并据此生成用户可见的濒危预警记录，记录预警等级（高危/中危/低危）以便前端筛选展示。

### 2.4.3　WDPA（全球保护地数据库）

WDPA（World Database on Protected Areas）是全球最全面的保护地信息数据库，收录了全球超过 27 万个保护地的地理边界、IUCN 保护类别、管理机构等信息[@reftodo007]。系统通过导入 WDPA 数据构建 `protected_areas` 表，支持按国家/地区代码（ISO3）、IUCN 保护类别、保护地类型等多个维度进行检索与统计展示。

### 2.4.4　WGSRPD（世界植物地理分区标准）

WGSRPD（World Geographical Scheme for Recording Plant Distributions）是由国际植物分类学协会（IAPT）制定的植物分布记录标准，将全球划分为约 369 个三级地理单元（Level-3 areas），为植物分布数据提供了标准化的空间描述框架[@reftodo008]。系统以 WGSRPD 三级分区代码存储植物分布数据（`plant_distributions` 表），并结合各分区的 GeoJSON 边界数据（`wgsrpd_regions_geo` 表），在前端渲染全球植物分布热力图。

### 2.4.5　Web 数据采集与接口逆向

在植物详情数据的补全过程中，系统采用了 Web 接口逆向分析技术。iPlant 植物智平台的详情页面（形态特征、生态习性、功用价值等内容）通过前端 JavaScript 动态加载，静态 HTML 中不包含上述字段，直接爬取页面 HTML 无法获取有效内容。

为此，通过分析页面加载的 JavaScript 文件（`plantinfo2.js`），发现了平台内部使用的 AJAX 数据接口 `/ashx/plantinfo.ashx`。该接口以 POST 方式接受物种编号（`spno`）和内容类型编号（`typeid`）为参数，`typeid=11` 对应形态特征，`typeid=12` 对应生态习性，`typeid=20` 对应功用价值。接口返回的 JSON 数组中，各子项的 `desc` 字段为 Base64 编码的中文文本，通过 `Buffer.from(desc, 'base64').toString('utf8')` 即可解码还原。

系统在采集过程中设置 1.5 至 2 秒的请求间隔以避免触发平台频率限制，并通过进度文件实现断点续传，保障长时间运行的稳定性。采集结果按"【子项名称】描述文本"格式写入 `plant_detail` 表，采用 `INSERT ... ON DUPLICATE KEY UPDATE` 语句确保已有手工录入内容不被覆盖。

## 2.5　本章小结

本章介绍了系统开发所采用的主要技术与数据标准。前端选用 React 19 + TypeScript + Vite + Tailwind CSS 构建组件化单页应用，并以浏览器原生 SVG 技术实现植物分布可视化；后端以 Node.js + Express + Sequelize 搭建 MVC 分层 API 服务，配合 JWT 实现无状态身份认证；数据库采用 MySQL 8.0，通过合理的索引设计支撑高频查询场景。在数据层，系统整合了 WCVP、IUCN、WDPA、WGSRPD 四个权威开放数据标准，并通过接口逆向分析技术补全植物详情内容。上述技术选型共同为系统的功能实现提供了支撑。

# 第三章　系统需求分析

## 3.1　功能需求分析

### 3.1.1　用户角色划分

本系统涉及两类前台用户角色：**游客**（未注册用户）和**注册用户**。两类角色在功能权限上存在差异，总体设计原则是游客可访问全部知识浏览与可视化功能，注册用户在此基础上解锁个性化的学习管理功能。

此外，系统后台存在**数据管理员**角色，主要通过脚本工具执行数据导入、中文名补全、图片资源标准化等数据维护操作，不通过前台界面提供服务，不在用户功能需求中展开说明。

### 3.1.2　游客功能需求

游客无需注册即可访问系统的全部知识展示功能，具体包括：

（1）**首页导览**：系统应提供统一的首页入口，展示平台定位、热门植物推荐和核心功能引导，使用户能够快速建立对系统结构的整体认知，并通过导航进入各功能模块。

（2）**植物库浏览与检索**（FR-01）：系统应支持植物列表的分页浏览、关键词搜索、科/属分类筛选和多维度排序（最新收录、热度、名称首字母），列表项应展示植物中文名、学名、封面图和基础分类信息。

（3）**植物详情查看**（FR-02）：用户点击植物列表项后，系统应展示完整的植物详情页，包括多图轮播展示、形态特征、生态习性、主要用途、地理分布描述，以及来自 IUCN 的濒危等级标注（如有）。

（4）**分类树浏览**（FR-03）：系统应支持科—属—种三级分类层级的交互式树形浏览，节点支持懒加载递归展开，叶节点（种级）可跳转至植物库进行筛选浏览。

（5）**全球分布热力图**（FR-04）：系统应以 WGSRPD 三级地理分区为单元，在交互式地图上以颜色深浅可视化植物物种分布密度，支持地图与表格视图切换，以及原生分布与引种分布的筛选。

（6）**数据统计展示**（FR-05）：系统应展示植物科属多样性统计和濒危等级分布等汇总数据，帮助用户从宏观角度理解植物资源特征。

### 3.1.3　注册用户功能需求

注册用户在游客功能基础上，额外具备以下个性化学习管理功能：

（1）**用户账号管理**（FR-06）：系统应支持用户注册、登录、查看个人资料，以及浏览数、收藏数、测验次数等学习统计信息的展示。

（2）**植物收藏**（FR-07）：用户可在植物详情页将感兴趣的植物添加至收藏列表，并在学习中心集中管理（含删除操作）。

（3）**浏览历史记录**（FR-08）：系统应在用户进入植物详情页时自动记录浏览事件，用户可在学习中心查看浏览历史。

（4）**知识测验**（FR-09）：系统应提供植物知识测验功能，用户完成答题后可查看得分、每道题的正误判断和题目解析，历史答题记录可在学习中心查看。

（5）**濒危预警管理**（FR-10）：系统应向注册用户推送 IUCN 濒危状态变化预警，用户可按预警等级和变化类型筛选预警列表，并对预警进行标记已读或忽略操作。

（6）**用户成就展示**（FR-11）：系统应在用户完成特定操作（如首次收藏、完成首次测验等）时解锁成就徽章，并在个人中心集中展示。

## 3.2　非功能需求分析

### 3.2.1　性能需求

系统在性能上应满足教学展示与功能演示的基本要求。考虑到系统植物数据规模较大（约 35 万条植物记录），高频访问接口应通过合理的索引设计控制查询耗时。参考性能目标如下：植物列表接口响应时间应控制在 800ms 以内；植物详情接口（含图片列表）应控制在 1000ms 以内；分类树顶层节点加载应控制在 500ms 以内；全球热力图地理数据接口因数据量较大，应控制在 2000ms 以内。

上述数值为参考指标，实际性能以测试章节的实测结果为准。

### 3.2.2　可用性需求

系统应支持主流现代浏览器（Chrome、Firefox、Edge 最新稳定版）的正常访问，并在 1024px 以上宽度的桌面端保持良好的布局显示效果。界面设计应简洁清晰，核心功能（植物检索、分类浏览、热力图查看）应在 3 步操作以内可达，降低新用户的学习成本。

### 3.2.3　安全性需求

系统应具备基本的安全保障能力。具体包括：用户密码采用 bcryptjs 进行哈希存储，禁止明文保存；需要用户身份的接口（收藏、答题、预警操作等）通过 JWT 中间件统一进行 Token 验证；后端启用 CORS 白名单，限制跨域请求来源；接口输入参数通过校验中间件进行格式检查，防止无效数据写入数据库。

### 3.2.4　可扩展性需求

系统架构应便于后续扩展。在代码组织上，采用 MVC 分层架构，业务逻辑封装于 Service 层，便于独立修改与替换；在数据模型上，`plant_detail.extra` JSON 字段支持在不修改表结构的前提下存储扩展属性；在数据工程上，数据导入与补全脚本独立于主服务运行，便于追加新数据源而不影响在线服务。

### 3.2.5　数据完整性需求

由于系统采用多源数据融合方式构建植物知识库，数据完整性尤为重要。系统通过数据库外键约束维护核心实体间的参照完整性（如 `plants.taxon_id → taxa.id`）；对于涉及主表与详情表的写入操作，采用数据库事务保证原子性；数据工程管道末端通过就绪度检查脚本（`check-data-readiness.js`）验证关键字段覆盖率（中文名覆盖、封面图覆盖、分类链完整性），确认达标后方对外提供服务。

## 3.4　本章小结

本章从功能需求和非功能需求两个层面对系统进行了分析。系统的核心目标是围绕植物科普场景，提供植物知识浏览、分类认知和数据可视化分析等主要功能，并通过学习交互提升用户参与度。功能需求按游客与注册用户两类角色分层设计，非功能需求在性能、可用性、安全性、可扩展性和数据完整性五个维度进行了说明。本章的需求分析结果将作为后续系统总体设计和功能实现的依据。
