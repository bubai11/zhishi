# 代码审查结果

## 1. 审查范围

基于对 `d:\zhishi-plant-system` 项目的真实代码扫描（包含前端组件、后端 API 路由、控制器、服务层、模型定义以及庞大的数据处理脚本目录），对系统当前的架构、功能、工程质量与数据状态进行了全面审查。本次审查严格对照“本科软件工程毕业设计标准”。

## 2. 架构问题

### 2.1 表现层“伪路由”问题
- **问题描述**：前端未采用标准的路由库（如 React Router），而是将几乎所有页面的切换、状态（如 `page`, `plantId`, `librarySearch` 等）集中在 `App.tsx` 中，通过手写 URL 解析和 `switch/case` 来挂载组件。
- **影响（系统/答辩）**：导致 `App.tsx` 逻辑臃肿，状态管理极其复杂且容易出错。在答辩中若被问及“前端路由的实现机制”或“应用状态如何管理”，很容易暴露工程不规范的问题。
- **是否影响答辩**：高
- **建议解决方案**：在答辩中应将其解释为“为了实现平滑的页面过渡和高度自定义的 URL 状态同步而设计的轻量级路由方案”，或承认其局限性并在“未来展望”中提出引入 React Router。

### 2.2 业务逻辑层过重（Fat Service）
- **问题描述**：后端采用了 MVC 变种（Controller -> Service -> Model），但在 `searchService.js` 等核心服务中，充斥了长达数百行的复杂原生 SQL 字符串拼接，且混合了缓存逻辑。
- **影响（系统/答辩）**：导致 Service 层既负责业务逻辑，又负责复杂的数据访问细节，一定程度上破坏了 ORM 带来的数据访问抽象层。
- **是否影响答辩**：中
- **建议解决方案**：在论文或答辩中，可将此行为包装为“在面对多源异构数据与复杂全文检索时，为追求极致性能（如 45 秒查询缓存）而采用的 SQL 定制优化与 QueryBuilder 模式”。

## 3. 功能完整性问题

### 3.1 缺失可视化的后台管理系统
- **问题描述**：后端存在 `routes/admin.js`，包含用于处理中文名人工审核、数据统计的接口，但前端完全没有对应的管理后台界面（如 Admin Dashboard）。
- **影响（系统/答辩）**：这是一个明显的“只留接口未落地”问题。系统声称具有“数据维护”能力，但实际是通过命令行脚本和零星接口完成的，缺乏系统化的管理闭环。
- **是否影响答辩**：高
- **建议解决方案**：在论文中必须明确界定“本系统专注于前台知识科普与学习展示”，将数据处理与维护定义为“线下数据工程管道”或“脚本支撑工具”，不要在论文中虚构“后台管理页面”。

## 4. 工程问题

### 4.1 “脚本替代系统”现象严重
- **问题描述**：`backend/scripts/` 和 `scripts/` 目录下存在高达 70+ 个脚本文件（涉及批量导入、数据映射、结构迁移、高优先级人工审核等）。系统的“多源数据融合”实际上是由这些离线脚本完成的，而非系统在线运行时的能力。
- **影响（系统/答辩）**：老师可能认为该系统只是一个“静态数据展示网站”，真正的核心工作（数据清洗）脱离了系统本体，工程集成度较低。
- **是否影响答辩**：高
- **建议解决方案**：在论文架构图中必须将“数据工程管道（Data Engineering Pipeline）”单独划分为一个脱机的基础设施层，将其作为系统的“前期准备工程”进行包装，凸显其作为“大数据清洗工具链”的价值。

## 5. 数据问题

### 5.1 数据维护能力高度依赖脚本操作
- **问题描述**：系统的动态更新能力极弱，当外部数据源（如 IUCN 红色名录）更新时，只能由开发者进入服务器执行特定脚本，不具备普通管理员的平滑操作能力。
- **影响（系统/答辩）**：对于一个软件工程项目，缺乏系统化的 CRUD 管理能力是一个短板。
- **是否影响答辩**：中
- **建议解决方案**：将这一缺陷合理化为设计妥协：“由于 WCVP、IUCN 等底层权威数据更新周期长、数据量极大，系统采用周期性离线数据工程重塑数据库，以保障线上系统检索性能和数据一致性。”

## 6. 总体风险结论

该系统整体完成度较高，前后台能够跑通，核心功能真实存在。但**最大的答辩风险在于“系统边界的描述”**。如果在论文中将脚本执行的数据融合过程描述为“系统的实时功能”，或者将未实现的后台管理描述为“系统功能模块”，将会在答辩中遭到致命质疑。必须将论文表述严格收敛于“前台可视化展示”+“后台提供服务”+“独立数据工程管道清洗数据”的三层逻辑。
